# Artist Pickup — 라이선스 후보 점수화·랭킹

아티스트 후보들을 받아 **6개 기준으로 점수화하고 랭킹**해서, "왜 이 아티스트인가"를 설명할 수 있는 shortlist를 만든다. 의사결정자가 읽는 결과물이므로 **결론(랭킹) 먼저, 근거 뒤에**.

프로젝트 원칙(→ 루트 `CLAUDE.md`)을 그대로 따른다: 코어는 안정·표현은 자유, **데이터 획득은 수단·방법을 가리지 않되 합법·ToS 범위 내**, **모든 점수엔 근거를**, 추정치를 사실처럼 쓰지 않는다.

## 입력
아티스트 풀(여러 명). 형태는 자유:
- 채팅에 이름 나열 / 이름 + 링크(Spotify·Instagram·YouTube URL이 있으면 정확도↑)
- 파일(csv/md/txt) 경로
- 사용자가 이미 아는 값(팔로워 수, 국적 등)을 인라인으로 주면 그대로 신뢰하고 출처를 "사용자 제공"으로 표기

목록이 모호하면(동명이인, 링크 없음 등) **임의로 추측하지 말고 사용자에게 확인**한다.

## 워크플로

### 1. 풀 파싱
아티스트 목록을 추출한다. 각 아티스트에 대해 가능하면 Spotify/IG/YouTube 식별자를 확보한다.

### 2. 데이터 획득 (하이브리드 — API에 갇히지 않는다)
**정확성 1순위 = 음악 / 스트리밍 / 세일즈 데이터.** 이게 가장 무겁게 쓰이고 틀리면 안 되는 데이터다. 여기에 가장 정확한 소스를 집중한다.

| 데이터 | 1차 소스 | 비고 |
|---|---|---|
| Spotify 팔로워·인기도·장르·카탈로그 수·라벨 | `scripts/spotify.py "이름"` | 키 없으면 `{"available": false}` 반환 → 웹서칭으로 대체 |
| 스트리밍 절대수치·월간청취자·앨범 세일즈 | 웹서칭 / 리서치 스킬 / 차트 벤더 | **공개 API에 없음.** 출처+신뢰도 필수, 못 찾으면 `null` |
| YouTube 구독자, IG 팔로워 | **info-site → 링크 추출 → insane-search** (아래 방법) | 맹목적 웹서칭보다 정확 |
| 국적·중국 활동 가능성·나이·인디/메이저 | 신뢰 정보처(나무위키·네이버·Wikipedia) + 리서치 스킬 (`insane-search`, `deep-research`, WebSearch) | 못 찾으면 "불명" |

규칙:
- **출처와 신뢰도를 항상 단다.** 예: `~5.4M monthly (Chartmetric, 중간 신뢰도)`, `IG 80만 (artist 공식계정, 높음)`.
- 확인 못 한 값은 **추정이라고 표기**하거나 `null`로 둔다. 빈 곳을 그럴듯하게 메우지 않는다.
- API는 한 수단일 뿐이다. Spotify 키가 없거나 막히면 웹서칭으로 같은 데이터를 찾는다.

#### 팔로워·기본정보 정확 획득법 (info-site → 링크 추출 → insane-search)
IG/YouTube 팔로워 수와 국적·나이 같은 기본정보는 검색 스니펫에 의존하면 부정확하다. 다음 순서로 *근거 있는* 값을 얻는다:

1. **신뢰 정보처에서 기본정보 + 공식 SNS 링크 추출.** 나무위키(`namu.wiki`), 네이버, Wikipedia 등에서 그 아티스트의 프로필을 읽고 **공식 Instagram·YouTube URL**을 뽑는다. (이런 위키형 사이트가 검색 스니펫보다 official handle을 정확히 담고 있다.)
2. **그 링크에 직접 접속해 실제 수치를 읽는다.** 막히면(`403`/`challenge`) `insane-search`의 엔진으로 우회:
   ```bash
   python3 -m engine "<instagram_or_youtube_url>"
   ```
   YouTube 구독자/채널 통계는 `yt-dlp --dump-json "<channel_url>"`로도 깔끔하게 받을 수 있다.
3. **출처와 신뢰도를 기록한다.** 예: `IG 1.2M (@handle, 공식계정 직접 확인, 높음)` vs `IG ~80만 (나무위키 기재, 중간)`. 접속·확인 못 하면 `null`로 두고 "미확인"으로 둔다 — 그럴듯한 숫자로 메우지 않는다.

> 왜: 정확성 1순위는 음악/스트리밍이지만, 팔로워도 "official 계정에서 직접 읽은 값"과 "스니펫 추정"은 신뢰도가 다르다. 이 차이를 점수가 아니라 *근거 문자열*에 남겨 의사결정자가 판단하게 한다.

`scripts/spotify.py`는 표준 라이브러리만 쓰므로 설치 불필요. 키는 `SPOTIFY_CLIENT_ID` / `SPOTIFY_CLIENT_SECRET` 환경변수(또는 export한 `.env`)로 받는다.

### 3. 채점 (6기준 × 0~100, 근거 포함)
아래 **루브릭**으로 각 아티스트의 6개 기준에 0~100 sub-score를 매긴다. 풀 전체를 보고 있으니 "풀 내 상대 위치"는 직접 판단한다(예: 이 풀에서 스트리밍 최고치를 90~100 앵커로 두고 나머지를 비례). 각 점수에 근거 한 줄을 붙인다. 데이터가 불명이면 그 기준은 `null`로 두고(점수에서 제외됨) 스코어카드에 표기한다.

### 4. 집계·랭킹 (스크립트)
sub-score를 JSON으로 모아 `scripts/score.py`에 넘긴다. 가중평균·티어·정렬·리포트 렌더를 **결정적으로** 처리한다(아래 "스크립트 사용").

### 5. 출력
채팅에 **랭킹 표 + 스코어카드**를 보여주고, `reports/artist-pickup-<라벨>.md`로 저장한다(`--report`). 끝에 한 줄 요약(누가 왜 1순위인지).

## 채점 루브릭

전부 **소프트 가중**이다 — 어떤 기준도 아티스트를 탈락시키지 않는다(하드 필터 없음). 비중은 기본값이며 사용자가 바꾸면 그 값을 쓴다.

기본 비중(높음=21% · 중간=14% · 낮음=7%):

| 기준 (key) | 비중 | 0~100 가이드 |
|---|:---:|---|
| `streaming` Spotify 스트리밍·지표 | **21%** | Spotify popularity(0-100, 절대치)와 월간청취자/스트리밍(풀 내 상대)을 블렌드. 둘 중 하나만 있으면 그것 기준. 둘 다 불명이면 `null`. |
| `genre` 장르 적합성 | **21%** | 힙합·R&B·POP = 85~100. 인접 장르(트랩·네오소울·일렉트로팝 등) = 60~80. 그 외 = 30~55. *배제는 아님.* |
| `label` 인디 vs 메이저 | **21%** | 인디펜던트/무소속 = 85~100(직접 계약 용이). 소형/디스트로 = 65~80. 메이저 레이블 = 35~55(레이블 경유 필요). 불명 = 55. |
| `followers` SNS 팔로워 | **14%** | **IG 50만↑ = 상위 등급(80~100).** 그 아래는 규모 + 풀 내 상대. IG 없으면 YouTube 구독자로 대체. |
| `catalog` 카탈로그 규모+반응 | **14%** | 발매 규모(풀 내 상대) × 시장 반응(평단·상업 반응 웹서칭). 신인이라 규모 작아도 반응이 강하면 상향 보정. |
| `china` 중국 활동 가능성 | **7%** | 중국 활동 가능/이력 있음 = 80~100. 불명/불확실 = 50. 활동 제한·불가 정황 = 10~30. (비중 낮음) |

**나이는 점수 항목이 아니다.** `stored.age`로 저장하고 스코어카드에 *타겟 오디언스 메모*로만 표시한다(노트 기준).

> 정규화 방식: 노트가 명시 기준을 준 항목(IG 50만, 장르 우선순위, 인디>메이저)은 **절대 앵커**로, 연속 수치(스트리밍·팔로워·카탈로그)는 **풀 내 상대**로 본다.

## 스크립트 사용

### Spotify fetch
```bash
python .claude/skills/artist-pickup/scripts/spotify.py "Artist Name"
```
`{"available": false}` 면 키가 없는 것 → 웹서칭으로 같은 필드를 채운다.

### 집계·랭킹·리포트
sub-score를 모아 JSON으로 만들고 `score.py`에 넘긴다. 입력 계약:

```json
{
  "label": "kpop-rnb-shortlist",
  "weights": {"streaming":0.21,"genre":0.21,"label":0.21,"followers":0.14,"catalog":0.14,"china":0.07},
  "artists": [
    {
      "name": "Artist A",
      "subscores": {"streaming":85,"genre":90,"label":80,"followers":70,"catalog":60,"china":50},
      "stored": {"age":27,"nationality":"USA"},
      "evidence": {
        "streaming":"Spotify popularity 68; ~5.4M monthly (Chartmetric, 중간)",
        "genre":"hip-hop/R&B (Spotify genres)",
        "label":"독립 (공식 SNS, 높음)",
        "followers":"IG 82만 (공식, 높음)",
        "catalog":"정규 2 + 싱글 11; 최근작 평단 호평",
        "china":"중국 활동 이력 불명 (낮음)"
      },
      "notes": "장르 적합 + 인디라 직접 계약 용이. 스트리밍도 풀 상위."
    }
  ]
}
```

실행 (랭킹 JSON 출력 + 리포트 저장):
```bash
python .claude/skills/artist-pickup/scripts/score.py input.json \
  --report reports/artist-pickup-<라벨>.md
```

`score.py`가 처리하는 것: 비중 정규화, **불명 기준 자동 제외 + 가중치 재정규화**, 가중평균 종합점수, 티어(S≥85 / A≥70 / B≥55 / C<55), 정렬, 마크다운 렌더. 산수·정렬은 사람이 다시 하지 않는다.

## 출력 형식

채팅에는 `score.py`가 만든 리포트와 같은 구조로 보여준다:

1. **랭킹 표** — 순위 · 아티스트 · 종합점수 · 등급
2. **스코어카드** (아티스트별) — 6기준 점수·근거 표 + 저장 항목(나이·국적) + 불명 기준 표기 + 맥락 코멘트
3. **한 줄 결론** — 1순위가 왜 1순위인지

신뢰도가 낮거나 불명인 데이터가 많으면 결과 위에 **"데이터 신뢰도 주의"**를 한 줄 명시한다. 정확성 1순위(스트리밍·세일즈)가 약하면 특히.
